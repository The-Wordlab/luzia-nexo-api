# Nexo Partner API Reference (Partner-Facing)

Status: living draft for partner onboarding.  
Audience: partner engineering teams integrating with Nexo from TypeScript or Python.

## 1) Integration model at a glance

Nexo supports two partner integration surfaces:

1. Inbound webhook mode
- Nexo calls your webhook when a user sends a message.
- Your server returns `{ "reply": "..." }` (or SSE stream in advanced mode).

2. Partner API mode
- Your backend calls Nexo endpoints to list subscribers, list threads, and post messages.
- Designed for proactive messaging and operational workflows.

## 2) Base URL and core endpoints

Base URL:
- `https://nexo.luzia.com/api` (environment-specific in staging/dev)

Core endpoints:
- `GET /apps/{app_id}/subscribers`
- `GET /apps/{app_id}/subscribers/{subscriber_id}/threads`
- `GET /apps/{app_id}/threads`
- `GET /apps/{app_id}/threads/{thread_id}/messages`
- `POST /apps/{app_id}/threads`
- `POST /apps/{app_id}/threads/{thread_id}/messages`
- `POST /apps/{app_id}/threads/{thread_id}/messages/assistant`

## 3) Authentication model (current truth)

### Primary partner auth (recommended)

Use both headers on every request:

- `X-App-Id: <app_uuid>`
- `X-App-Secret: <plaintext_secret>`

Why both are required:
- `X-App-Secret` is stored hashed (bcrypt-style) in backend storage, so secret-only lookup is not practical.
- `X-App-Id` identifies which app secret hash to verify against.
- This is the clean and intentional partner credential pair.

### Dashboard user auth (internal/operator usage)

- `Authorization: Bearer <dashboard_user_jwt>`
- This is for org members using dashboard-connected flows, not ideal for external server-to-server partner integrations.

### OAuth client credentials endpoint (exists, but not primary runtime path yet)

- `POST /api/oauth/token` exists and issues JWTs from app credentials.
- Current partner route authorization dependency primarily supports:
  - app header auth (`X-App-Id` + `X-App-Secret`), or
  - dashboard user JWTs.
- Treat OAuth client-credentials token flow as an evolution path until route-level acceptance semantics are fully aligned and documented.

## 4) App ID and secret lifecycle

In dashboard create/edit app screens:

- App ID is stable and read-only.
- Webhook secret is set/generated by operator.
- Secret is masked after save.
- Clearing or regenerating secret rotates credentials and can break existing integrations unless partner config is updated.

Operational recommendation:

1. Generate secret in dashboard.
2. Copy once into partner secret manager.
3. Verify integration with test endpoint.
4. Rotate with overlap window when possible (future improvement: dual-active secret support).

## 5) Webhook contract

When your app is configured in webhook mode, Nexo POSTs to your webhook URL when a user sends a message.

### Request format

Nexo sends a JSON POST with headers:

- `Content-Type: application/json`
- `X-App-Id: <app_uuid>`
- `X-Timestamp: <unix_seconds>` (when signing is enabled)
- `X-Signature: sha256=<hex_digest>` (when signing is enabled)
- `X-Trace-ID: <uuid>` (for correlation)

Request body:

```json
{
  "message": "user message text",
  "thread_id": "uuid",
  "user_id": "uuid",
  "profile": { "locale": "en", "name": "User Name", ... }
}
```

### Response format (synchronous)

Return HTTP 200 with JSON:

```json
{
  "text": "assistant reply text",
  "metadata": { "custom_key": "optional" },
  "content_parts": [{ "type": "text", "text": "optional structured parts" }]
}
```

The `text` field is canonical. Legacy `reply` field is also accepted as fallback.

### Response format (SSE streaming)

Return HTTP 200 with `Content-Type: text/event-stream` and SSE events:

```
data: {"type": "delta", "text": "partial "}
data: {"type": "delta", "text": "response"}
data: {"type": "done", "text": "partial response"}
```

## 6) Request signing (HMAC-SHA256)

When a webhook secret is configured, Nexo signs every outgoing request so partners can verify authenticity.

### Signing process

1. Generate unix timestamp (integer seconds)
2. Build signed payload: `"{timestamp}.{raw_json_body}"`
3. Compute `HMAC-SHA256(signing_secret, signed_payload)`
4. Send as headers: `X-Timestamp: {timestamp}`, `X-Signature: sha256={hex_digest}`

### Verification (partner side)

```python
import hmac, hashlib, time

def verify_signature(secret, timestamp, body, signature, max_age=300):
    # Reject stale requests
    if abs(time.time() - int(timestamp)) > max_age:
        return False
    signed_payload = f"{timestamp}.{body}"
    expected = "sha256=" + hmac.new(
        secret.encode(), signed_payload.encode(), hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature)
```

## 7) Error codes and status handling

### Nexo-to-partner (webhook calls)

| Your response | Nexo behavior |
|---|---|
| 200 + valid JSON | Success, reply delivered to user |
| 200 + invalid JSON | Error surfaced to user |
| 4xx | Not retried, error surfaced |
| 500, 502, 503, 504 | Retried with exponential backoff |
| Timeout (>8s default) | Retried, then error surfaced |

### Partner-to-Nexo (API calls)

| Status | Meaning |
|---|---|
| 200 | Success |
| 401 | Invalid or missing credentials |
| 403 | Insufficient permissions (wrong org) |
| 404 | Resource not found |
| 422 | Validation error (check `detail` field) |
| 429 | Rate limited (respect `Retry-After` header) |

## 8) Retry semantics

Nexo retries webhook calls on transient failures:

- Retryable: HTTP 500, 502, 503, 504, and timeouts
- Not retried: HTTP 4xx (client errors)
- Backoff: exponential, base 0.5s, max 8s delay
- Default max retries: 0 (configurable per app)
- Timeout: 8 seconds per attempt (configurable)
